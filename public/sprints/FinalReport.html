  1. Donate or support? Which wording increases conversions through the donation link text?

There was a bit of a schism within our team when it came to this last measure. We were thrown a curveball and couldnt effectively figure out how to make this work. Two of our team members went to TAs at different times within the previous two sprints to try and figure out how to properly implement the analytics piece into our application. In one of the sessions that occurred last week, our team member worked with a TA to set up the RSVP text experiment as well as what at least one member of our team believed was the donate vs. support experiment. The RSVP experiment was successfully implemented, but a lack of results for the donate vs. support experiment gave us reason for a bit of pause. As a result, the TA decided to add two extra experiments to as a way of seeing if they would perhaps produce results vs. our previous attempt. To this day, for all three experiment attempts, weve only gotten one successful conversion, out of hundreds of sessions, which has led one of our team members (not the member in charge of analytics), to believe that the data we have for this section is insufficient.

Its more effective to leave some ambiguity within a given page to first draw the consumer in, so that you have the ability to slowly try and win them over, convert people who wouldnt consider donating before to donate. Thats why, in general, its better to have something thats more open-ended, especially near the top of the page, to draw people in. The word donate scares people off too quickly.

Project Maintenance

Here, well describe how we would move forward from our current application, which implements all of the features that had been described within the project spec.

There are three next steps that need to be taken, if our application would begin to see live traffic. First and most importantly, a persistent data store is necessary. Currently, we store our events and all data about them within our projects main memory (the models/events.js file is where the events currently reside, stored as an array of dictionaries). From this current setup, we can very easily roll over into a MongoDB database. MongoDB, which stores JSON documents, is a very popular NoSQL database, mainly because of its ease of use and its strong integration with Javascript objects.

To successfully implement the database, we can set up an Restless Events database with an events collection. To put our events into the database, it would require inserting the Javascript object for each eventthat is, loop through events and insert using something similar to: for (event in events){ db.events.insert(event);}. If we were to do this, we would need to make an API of sorts for the application to deal with the events data (this would be a file, such as eventsDAO.js), from we could abstract the existence of the database, which is an advantage of the MVC model.

The second step would be to create a more cohesive design. Our team is composed of two SOM students, one East Asian Studies major, and a Math/CS major, so none of us have much advanced design experience. An overall design would require much more thinking of the product and the core experiencethat is, how we want to see our application and how we want to present it. For example, the logo for Eventbrite itself is meticulously crafted, as it seems like a ticket stub. There is no doubt a great deal of market insight behind every single design decision that a professionally developed application such as Eventbrite has made. From the colors to the images that a user sees, the experience of using an app should feel complete, and that is something that our app lacks as of yet. Nevertheless, for the design experience we all had, the application we present has a solid initial design that could hopefully be built upon on the road to a more complete user experience.

The third step would be improve the feature set in our application and solidify what experience we are offering. Obviously, we would need to set ourselves apart from other event planning websitesmost notably, Eventbrite. This would be done by viewing the business analytics and seeing what our web application is being used for. Then, we would, in a way, pivot by improving our features in that specific direction and growing to support the use cases our users want to see.

To do these, we would need to roll this off to another team. With common wisdom and the knowledge gained from this semester, we see this as an interesting challenge and have a few ideas as to how to approach it. We need to sit down with the future team and explain the feature set (or show the spec we were provided), explain how our code is broken downi.e. the Model-View-Controller formatand comment it to ease understanding. We should also do a thorough demo for the team to explain functionality. In the end, there is a point where we will simply need to let go of the project and let the next team take over. All readings on the matter to conclude the same points. For example, see http://stackoverflow.com/questions/1377184/how-to-hand-over-a-project-systematically and http://cobaltpm.com/project-hand-off/.

Lessons Learned

Overall, through building Restless Events, we all received a good look into what goes into making a formal web application, from the development to the design through to the business analytics. It has been a good experience for all members of a team, not to mention a nice dive into the world of programming.

The core framework of the project provided a valuable overview of github and the various other applications that support development through github. In particular, our team enjoyed the use of waffle.io as an easy way to organize work flow and have a good overview on the status of all the user stories. There is a very positive sense of satisfaction of moving something into the done column, and that in and of itself might be strong argument to use a kanban board when working in a team. As restless-paper began to tackle user stories, we also learned about some of the frustrations that might occur from tasks that take variable amounts of time to complete. We knew going in that the point system we used would be a rough estimate, but some of the stories had completion times that were very different from expectations. In particular, we have spent a considerable amount of time chasing bugs. This troubleshooting process can often take as long as implementation of the actual features, and in the future, it may be prudent to include troubleshooting time in the point estimation for the various user stories. When we asked ourselves at the beginning of the project how long will this feature take to build?, we really were answering a different question from how long will this feature take to get working?. This project also might be taken as a lesson of when to step away from a problem, and seek help from those who might have an answer. The weekly sessions and office hours this class offered were a tremendous resource.

Many members of the team were completely new to the world of programming. For those members, learning to become familiar with using github was one of the largest lessons in and of itself. Its not immediately intuitive to the outside world why the architecture of a project is designed the way it is. In the beginning, the process of pulling to separate screens, then to separate branches seems redundant and frustrating. Oftentimes, it seems more logical to introduce a UI for this type of work due to the unfamiliarity with the commands. As the project grew, however, it became easy to see how many of the frustrations were due to our project being rather limited in scope. There were many moments in which those of us unfamiliar with code finally gained a true appreciation for why things are separated when collaborating with others, and an understanding of how things look when scaled up to hundreds or even thousands of separate developers.

The group as a whole placed higher value on code organization as the project progressed. From redundant lines of code that could be pulled from the common layout theme all the way to overlapping styles in CSS, we learned the hard way that losing a sense of housekeeping can break the program entirely, and that it is time consuming to go back and correct.

Overall, it provided great insight into what makes the applications we use every day tick. This might be extended to the learnings of the class as a whole, but the opportunity to actually apply it is something of a capstone to the course. When looking at the start-up page to our favorite mobile phone app to the various input fields of a registration page, it becomes more natural to visualize where the things you are seeing come from, where the things you are typing go to, how the messages flow from user to server, and the lines of code that made it happen.

This has been a great experience.
